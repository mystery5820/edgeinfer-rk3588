# Phase 5：YOLOv11 压缩路线实验结论

## 背景

本阶段尝试对 YOLOv11n 进行进一步模型压缩，目标是在 RK3588 端侧部署场景下减少模型参数量、降低计算量并提升 NPU 推理效率。

## 已验证方案

### 1. torch-pruning 自动结构化剪枝

尝试使用 torch-pruning 的 MagnitudePruner 对 YOLOv11n 进行结构化通道剪枝。实验结果显示，Pruned groups 为 0，剪枝前后参数量没有变化。

结论：当前 Ultralytics YOLOv11 结构与 torch-pruning 自动剪枝模式兼容性不足，无法生成有效剪枝组。

### 2. 手动 DependencyGraph 剪枝探测

进一步尝试手动指定 Conv2d 层进行依赖图剪枝探测。实验中多个候选 Conv2d 层均无法被识别为有效依赖图节点，最终未找到可安全剪枝组。

结论：后训练结构化剪枝路线在当前模型结构上不稳定，不适合作为主线继续推进。

### 3. width0875 宽度压缩

构建 width0875 版本 YOLOv11n，将模型宽度压缩到 baseline 的约 87.5%。该模型参数量从约 2.62M 降至约 2.08M，压缩幅度约 20%。

但由于通道结构发生变化，预训练权重只能部分迁移。短周期 fine-tune 后 mAP 基本无法恢复。

结论：width0875 压缩幅度较明显，但精度恢复困难，不适合作为当前最终模型。

### 4. width096875 极轻量宽度压缩

构建 width096875 版本 YOLOv11n，参数量从 2,624,080 降至 2,539,652，参数减少约 3.2%。

结论：该方案压缩幅度过小，预计对 RK3588 NPU 推理速度和模型体积提升有限，不值得作为正式压缩路线。

## 最终决策

YOLOv11n 本身已经是轻量级检测模型，在当前项目中继续进行结构化剪枝的收益有限，且会带来明显的精度恢复成本和工程不确定性。

因此，视觉模型最终采用：

- YOLOv11n baseline；
- baseline fine-tune 版本作为训练参考；
- RKNN INT8 量化模型作为端侧部署模型；
- RK3588 板端真实 Benchmark 作为最终性能评估依据。

## 后续工作

1. 导出 YOLOv11n baseline fine-tune 权重为 ONNX；
2. 转换 RKNN INT8；
3. 在 RK3588 上运行真实 NPU Benchmark；
4. 生成端侧性能报告；
5. 将无效剪枝实验作为模型压缩探索过程记录，而不作为最终部署模型。
